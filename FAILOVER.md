# 分散ステート・フェイルオーバー方針 (Phase 1.8)

## スプリットブレイン対応方針
本システムは `libcluster` (Gossip戦略) と `Horde` を用いてセッションレジストリを共有しています。
ネットワーク分断（スプリットブレイン）が発生した場合、Horde の CRDT ベースのレジストリは一時的に各パーティションで独立して動作しますが、ネットワーク回復時に自動的にマージされます。
ただし、同一セッションへの操作が両側で発生するコンフリクトを避けるため、Eliterm では「セッションは特定の一つのノードのローカルディスク(`home/`)に物理的に紐付いている」という制約を設けています。
したがって、ネットワーク分断時でも、対象の `home/` ディレクトリを持たないノードからそのセッションを起動・操作することはできず、論理的なコンフリクトは自然に防がれます。

## ノードダウン検知時のリカバリフロー
一般的な Erlang/OTP や Horde のユースケースでは、ノードダウン時に別ノードでプロセスを自動再起動（フェイルオーバー）させます。
しかし Eliterm では、PTY に紐付く bash セッションのデータ（`home/` ディレクトリ、スクリプト、ログ）が各ノードのローカルディスクに保存されており、SAN等の共有ストレージを前提としません。

そのため、以下のポリシーを採用します：
1. **自動フェイルオーバーの無効化**:
   あるノードがダウンした場合、Horde による別ノードでの `Eliterm.ShellSession` の自動再起動は原則行いません。（ダウンしたノードが復帰するのを待つか、事前に `migrate` を行う必要があるため）。
2. **ダウンノードの復旧**:
   ダウンしたノードが再起動した場合、ローカルの `~/.eliterm/sessions/` を走査し、自分に紐付くセッション群を再起動（Quantum スケジュールや PTY の再開）する自己復旧処理を行います。
3. **強制的なオフライン移行（将来構想）**:
   ノードが物理的に完全に破損した場合、バックアップアーカイブ等から別ノードへ手動で復元し、Horde レジストリに再登録する運用ツールを将来的に提供します。
