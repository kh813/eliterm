# Antigravity エージェント運用ガイドライン

`eliterm` プロジェクトにおける作業は、以下のルールに厳密に従って進めること。

## 1. 参照すべき重要ファイル
- **仕様書**: `eliterm_specs.md`
- **作業タスクリスト**: `eliterm_todo.md`
作業を進めるにあたり、常にこれらのファイルを参照し、要件と現在のタスク状況を把握すること。

## 2. タスク管理とフェーズ完了のワークフロー
- **タスクの完了**: 一つのタスクが終了したら、必ず `eliterm_todo.md` を編集して該当タスクを完了状態（チェックマークをつける等）にすること。
- **テスト・ビルドの徹底と不具合修正**: `git commit` を行う前、または GitHub Actions をトリガーする（`git push --tags` など）前には、必ずローカルでビルドやテストコード（`mix compile --warnings-as-errors` や `mix test` 等）を実行し、エラーやワーニングが一切出ないことを確認すること。
- **UI/UX変更時のローカル動作テストの徹底（必須）**: xterm.jsのレイアウト調整やLiveViewのマークアップ変更など、表示やUIに関わる修正を行った場合、本番リリース（タグを打ってプッシュするなど）を行う前に、必ずユーザーに対してローカル環境での起動テスト（`mix run --no-halt` や `iex -S mix`）を依頼すること。目視によるレイアウトの崩れや無限リサイズループなどのデグレが発生していないかを確認してもらい、テストパスの返答を得てから初めてリリースプロセスに進むこと。
- **バージョン番号の無断更新禁止**: リリース時のバージョン番号（Gitタグなど）をエージェントの独断で勝手に進めないこと。必ずユーザーに対して使用すべきバージョン番号を事前に確認し、ユーザーの指定した番号でのみコミット・タグ打ちを行うこと。
- **コミットとフェーズ完了**: 上記のテスト・ビルド・ローカル動作確認を通過してフェーズが完了したら、必ず `git commit` を行い、作業の区切りをつける。その後、次のフェーズへ進むこと。

## 3. 作業ログの記録
- **DEVLOG.md の保守**: 作業の過程、発生した問題、技術的な決定事項、現在の進捗などを `DEVLOG.md` に適宜記録すること。これにより、他のエージェントや将来のセッションへの引き継ぎをスムーズに行う。

## 4. コミュニケーションと意思決定のサポート
- **確認事項の報告**: 仕様書に矛盾がある場合や、実装にあたって確認が必要な不明点が生じた場合は、独断で進めずにユーザーへ質問すること。
- **選択肢の提示**: ユーザーに判断を仰ぐ際は、判断しやすいように複数の選択肢を提示し、それぞれのメリット・デメリットをあわせて説明すること。
